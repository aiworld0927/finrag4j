# Prompt整理 - 部署与接口优化阶段

## 步骤十七：RocketMQ配置问题排查

### 原始提示词
```
agent@christ:~/finrag4j$ docker-compose -f docker-compose-base.yml up -d 
 [+] up 7/7 
  ✘ Container finrag4j-rocketmq-namesrv   Error dependency rocketmq-namesrv failed to start  
  dependency failed to start: container finrag4j-rocketmq-namesrv is unhealthy
```

### 问题分析

1. **NameServer启动成功但healthcheck失败**：
   - 日志显示：`The Name Server boot success`
   - healthcheck使用`nc -z localhost 9876`检测失败
   - 原因：RocketMQ 5.x镜像中可能没有`nc`命令

2. **内存资源限制**：
   - RocketMQ默认需要较大内存
   - 容器可能因内存不足而无法正常启动

### 解决方案

**1. 修改healthcheck检测方式**：
```yaml
healthcheck:
  test: ["CMD-SHELL", "cat /home/rocketmq/logs/namesrv.log | grep -q 'boot success' && exit 0 || exit 1"]
  interval: 15s
  timeout: 10s
  retries: 10
  start_period: 30s
```

**2. 配置小内存**：
```yaml
environment:
  # NameServer
  - JAVA_OPT=-Xms256m -Xmx512m -Xmn128m
  # Broker
  - JAVA_OPT=-Xms512m -Xmx1g -Xmn256m

deploy:
  resources:
    limits:
      memory: 512M  # NameServer
      # memory: 1G   # Broker
    reservations:
      memory: 256M  # NameServer
      # memory: 512M # Broker
```

### 内存配置总结

| 服务 | JVM初始堆 | JVM最大堆 | 容器内存限制 |
|------|---------|---------|-------------|
| NameServer | 256MB | 512MB | 512MB |
| Broker | 512MB | 1GB | 1GB |

## 步骤十八：Python服务接口实现

### 原始提示词
```
python需要实现接口，包括不限于 
 1. GET /health — 服务健康心跳检测，适配Nacos健康巡检 
 2. POST /api/parse/file — 通用文档解析（PDF/Word/Excel/TXT），返回纯文本、页数、文件类型 
 3. POST /api/parse/ocr — 离线PaddleOCR识别，支持图片/扫描PDF，自带图像去噪矫正，返回识别文本、平均置信度 
 4. POST /api/text/clean — 金融专属文本清洗，去除页眉页脚、水印、冗余空行、制式垃圾文本 
 5. POST /api/text/split — 金融三策略智能分块（regulatory监管文件/credit信贷合同/official公文），支持自定义分块大小、重叠度
```

### 接口清单

| 接口路径 | HTTP方法 | 功能描述 |
|---------|---------|---------|
| `/health` | GET | 健康检查（适配Nacos） |
| `/api/parse/file` | POST | 通用文档解析 |
| `/api/parse/ocr` | POST | 离线OCR识别 |
| `/api/text/clean` | POST | 文本清洗 |
| `/api/text/split` | POST | 文本分块 |

### 接口详细说明

**1. GET /health**
```json
{
  "status": "healthy",
  "service": "finrag4j-python",
  "timestamp": "2024-01-15T10:30:00",
  "version": "1.0.0",
  "message": "服务运行正常"
}
```

**2. POST /api/parse/file**
```json
{
  "success": true,
  "file_type": "pdf",
  "text": "解析的纯文本内容...",
  "page_count": 10,
  "message": "解析成功"
}
```

**3. POST /api/parse/ocr**
```json
{
  "success": true,
  "text": "识别的文本内容...",
  "confidence": 0.95,
  "page_count": 5,
  "message": "识别成功"
}
```

**4. POST /api/text/clean**
```json
{
  "success": true,
  "text": "原始文本摘要...",
  "cleaned_text": "清洗后的文本...",
  "cleaned_length": 1500,
  "original_length": 2000,
  "message": "清洗成功"
}
```

**5. POST /api/text/split**
```json
{
  "success": true,
  "chunks": [
    {"content": "块1内容", "chunk_index": 0, "start_pos": 0, "end_pos": 500}
  ],
  "strategy": "regulatory",
  "total_chunks": 10,
  "message": "分块完成"
}
```

### 分块策略

| 策略名称 | 适用场景 | 默认块大小 | 重叠度 |
|---------|---------|-----------|-------|
| `regulatory` | 监管文件、法律法规 | 600字符 | 150字符 |
| `credit` | 信贷合同、担保协议 | 800字符 | 200字符 |
| `official` | 公文、内部通知 | 400字符 | 80字符 |

### 新增文件

| 文件 | 功能 |
|------|------|
| `services/text_cleaner.py` | 文本清洗服务 |

### 修改文件

| 文件 | 修改内容 |
|------|---------|
| `main.py` | 实现5个API接口 |
| `services/__init__.py` | 导出text_cleaner |
| `services/text_chunker.py` | 更新策略名称 |

## 关键技术要点

### 1. Docker Compose配置分离
- 使用`include`语法引用基础配置
- 中间件与应用服务分离管理
- 支持独立启动中间件或应用

### 2. 环境变量管理
- 使用`.env`文件统一管理配置
- 支持多环境切换（dev/test/prod）
- 敏感信息不硬编码

### 3. 健康检查策略
- 使用日志检查替代端口检测
- 设置合理的启动等待时间
- 配置适当的重试次数

### 4. 内存资源管理
- 根据服务类型配置不同内存限制
- 避免资源浪费和OOM问题

### 5. 接口规范
- 统一响应格式（success/message/data）
- 明确的请求参数和返回结构
- 适配Nacos服务发现
