## 1. 日志规范

### 1.1 日志级别定义

| 级别 | 用途 | 示例 | 生产环境输出 |
|------|------|------|-------------|
| ERROR | 系统错误，需要立即处理 | 数据库连接失败 | 是 |
| WARN | 警告，潜在问题 | 请求参数不合法 | 是 |
| INFO | 关键业务事件 | 用户登录、文档上传 | 是 |
| DEBUG | 调试信息 | SQL语句、请求详情 | 否 |
| TRACE | 详细跟踪 | 方法入参出参 | 否 |

### 1.2 日志格式（JSON）

```json
{
  "timestamp": "2026-06-25T10:30:00.123+08:00",
  "level": "INFO",
  "logger": "com.finrag4j.auth.controller.AuthController",
  "thread": "http-nio-8081-exec-3",
  "traceId": "a1b2c3d4e5f6",
  "spanId": "a1b2c3d4",
  "tenantId": 1,
  "userId": 10001,
  "message": "用户登录成功",
  "method": "POST /auth/login",
  "duration": 45,
  "status": 200,
  "clientIp": "192.168.1.100",
  "metadata": {
    "username": "admin",
    "loginType": "password"
  }
}
```

### 1.3 日志内容规范

**必须包含的字段：**
- traceId：链路追踪ID
- tenantId：租户ID（多租户系统）
- userId：用户ID
- 操作结果：成功/失败
- 耗时：接口/方法执行耗时(ms)

**禁止记录的内容：**
- 密码、Token、密钥
- 身份证号、银行卡号（或脱敏后记录）
- 完整的SQL参数（含敏感信息时）

### 1.4 各模块日志规范

#### Java 后端（Logback）

```xml
<!-- logback-spring.xml -->
<configuration>
    <property name="LOG_PATTERN" 
        value='{"timestamp":"%d{ISO8601}","level":"%level","logger":"%logger","thread":"%thread","traceId":"%X{traceId}","message":"%msg"}%n'/>
    
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="net.logstash.logback.encoder.LogstashEncoder"/>
    </appender>
    
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>/var/log/finrag4j/app.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>/var/log/finrag4j/app.%d{yyyy-MM-dd}.%i.log</fileNamePattern>
            <maxHistory>30</maxHistory>
            <maxFileSize>100MB</maxFileSize>
        </rollingPolicy>
        <encoder class="net.logstash.logback.encoder.LogstashEncoder"/>
    </appender>
    
    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="FILE"/>
    </root>
</configuration>
```

#### Python 服务

```python
# logging_config.py
import logging
import json
from pythonjsonlogger import jsonlogger

class CustomJsonFormatter(jsonlogger.JsonFormatter):
    def add_fields(self, log_record, record, message_dict):
        super().add_fields(log_record, record, message_dict)
        log_record['traceId'] = getattr(record, 'traceId', 'unknown')
        log_record['tenantId'] = getattr(record, 'tenantId', None)
        log_record['service'] = 'python-service'

formatter = CustomJsonFormatter(
    '%(timestamp)s %(level)s %(name)s %(message)s'
)
```

## 2. 监控体系

### 2.1 监控维度

| 维度 | 工具 | 指标 |
|------|------|------|
| 基础设施 | Prometheus + Node Exporter | CPU、内存、磁盘、网络 |
| 应用性能 | Micrometer + Prometheus | JVM、HTTP请求、数据库 |
| 业务指标 | Prometheus + Grafana | 用户数、文档数、问答数 |
| 链路追踪 | Jaeger/SkyWalking | 请求链路、耗时分析 |
| 日志分析 | ELK Stack | 错误日志、异常统计 |
| 服务状态 | Nacos + Sentinel | 服务健康、限流熔断 |

### 2.2 关键监控指标

#### 系统层面

| 指标 | 采集方式 | 告警阈值 |
|------|---------|---------|
| CPU使用率 | Prometheus | > 80% |
| 内存使用率 | Prometheus | > 85% |
| 磁盘使用率 | Prometheus | > 90% |
| 网络带宽 | Prometheus | > 80% |
| 系统负载 | Prometheus | > CPU核数*2 |

#### 应用层面

| 指标 | 采集方式 | 告警阈值 |
|------|---------|---------|
| JVM堆内存 | Micrometer | > 80% |
| GC次数/分钟 | Micrometer | > 10次 |
| 线程数 | Micrometer | > 500 |
| HTTP错误率 | Micrometer | > 1% |
| HTTP P99延迟 | Micrometer | > 5s |
| 数据库连接池 | Micrometer | > 80% |

#### 业务层面

| 指标 | 采集方式 | 说明 |
|------|---------|------|
| 活跃用户 | 应用埋点 | 日/周/月活 |
| 文档处理量 | 应用埋点 | 上传/解析/检索 |
| 问答成功率 | 应用埋点 | 成功回答/总请求 |
| 向量检索延迟 | 应用埋点 | P50/P95/P99 |

## 3. 告警规则

### 3.1 告警级别

| 级别 | 响应时间 | 通知方式 | 场景 |
|------|---------|---------|------|
| P0-紧急 | 5分钟 | 电话+短信+邮件 | 服务宕机、数据丢失 |
| P1-严重 | 15分钟 | 短信+邮件+Slack | 核心功能不可用 |
| P2-警告 | 1小时 | 邮件+Slack | 性能下降、资源紧张 |
| P3-提示 | 4小时 | Slack | 非关键异常 |

### 3.2 告警规则配置

```yaml
# Prometheus alerting rules
groups:
  - name: finrag4j-alerts
    rules:
      # 服务宕机
      - alert: ServiceDown
        expr: up{job=~"finrag4j-.*"} == 0
        for: 1m
        labels:
          severity: P0
        annotations:
          summary: "服务 {{ $labels.job }} 宕机"
          description: "服务 {{ $labels.instance }} 已宕机超过1分钟"
      
      # 错误率过高
      - alert: HighErrorRate
        expr: |
          sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m]))
          /
          sum(rate(http_server_requests_seconds_count[5m])) > 0.01
        for: 2m
        labels:
          severity: P1
        annotations:
          summary: "HTTP错误率过高"
          description: "错误率 {{ $value | humanizePercentage }} 超过1%"
      
      # 响应时间过高
      - alert: HighLatency
        expr: |
          histogram_quantile(0.99,
            sum(rate(http_server_requests_seconds_bucket[5m])) by (le, uri)
          ) > 5
        for: 5m
        labels:
          severity: P1
        annotations:
          summary: "接口 {{ $labels.uri }} P99延迟过高"
          description: "P99延迟 {{ $value }}s 超过5秒"
      
      # JVM内存不足
      - alert: JvmMemoryHigh
        expr: |
          jvm_memory_used_bytes / jvm_memory_max_bytes > 0.85
        for: 5m
        labels:
          severity: P2
        annotations:
          summary: "JVM内存使用率过高"
          description: "内存使用率 {{ $value | humanizePercentage }}"
      
      # 磁盘空间不足
      - alert: DiskSpaceLow
        expr: |
          (node_filesystem_avail_bytes / node_filesystem_size_bytes) < 0.1
        for: 5m
        labels:
          severity: P1
        annotations:
          summary: "磁盘空间不足"
          description: "磁盘 {{ $labels.device }} 剩余空间 {{ $value | humanizePercentage }}"
```

### 3.3 告警收敛

```yaml
# Alertmanager 配置
global:
  smtp_smarthost: 'smtp.example.com:587'
  smtp_from: 'alert@finrag4j.com'

route:
  group_by: ['alertname', 'severity']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  receiver: 'default'
  routes:
    - match:
        severity: P0
      receiver: 'p0-team'
      group_wait: 10s
      repeat_interval: 5m
    - match:
        severity: P1
      receiver: 'p1-team'
      repeat_interval: 30m

inhibit_rules:
  - source_match:
      severity: 'P0'
    target_match:
      severity: 'P1'
    equal: ['instance']
```

## 5. 链路追踪

### 5.1 追踪规范

```java
// 自动配置（Spring Cloud Sleuth + Micrometer Tracing）

```

### 5.2 关键追踪点

| 追踪点 | 说明 |
|--------|------|
| Gateway入口 | 请求进入系统 |
| Auth鉴权 | JWT验证耗时 |
| 业务处理 | 核心逻辑执行 |
| 数据库访问 | SQL执行耗时 |
| 缓存访问 | Redis读写耗时 |
| 外部调用 | Python服务/大模型API |
| Gateway出口 | 响应返回 |
