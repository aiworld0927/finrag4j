# 44_代码审查规范

## 1. 审查流程

### 1.1 PR提交要求

- 每个PR只解决一个问题或实现一个功能
- PR标题清晰描述改动内容
- 提交信息遵循约定式提交规范
- 附带相关Issue链接

### 1.2 审查步骤

1. **自动检查**：CI/CD自动运行代码格式化、静态检查、单元测试
2. **人工审查**：至少1位代码所有者审查
3. **反馈修正**：根据审查意见修改代码
4. **合并确认**：所有审查通过后合并

## 2. 审查要点

### 2.1 代码质量

| 检查项 | 标准 |
| :--- | :--- |
| 代码可读性 | 变量命名清晰，注释充分 |
| 复杂度 | 方法行数不超过50行，避免嵌套过深 |
| 重复代码 | 无明显重复逻辑，可复用部分已抽取 |
| 错误处理 | 异常捕获完整，错误信息清晰 |

### 2.2 安全性

| 检查项 | 标准 |
| :--- | :--- |
| SQL注入 | 使用预编译语句或ORM框架 |
| XSS攻击 | 前端输入进行转义处理 |
| 敏感信息 | 密码等敏感信息不打印日志 |
| 权限控制 | 接口访问权限校验完整 |

### 2.3 性能

| 检查项 | 标准 |
| :--- | :--- |
| 数据库查询 | 避免N+1查询，合理使用索引 |
| 缓存使用 | 热点数据已缓存，缓存策略合理 |
| 并发处理 | 线程安全，资源释放及时 |

### 2.4 测试覆盖

- 新增代码单元测试覆盖率≥80%
- 核心业务逻辑有集成测试
- 无测试用例的代码需要说明原因

## 4. 审查责任

- **作者**：确保代码符合规范，编写测试用例
- **审查者**：客观公正评估，提供建设性意见
- **维护者**：确保审查流程执行到位

## 5. 合并条件

- 至少1位代码所有者批准
- 所有自动化测试通过
- 无未解决的严重问题
- 代码已根据审查意见修改完成
