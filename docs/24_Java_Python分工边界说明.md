# FinRag4j Java/Python分工边界说明

## 文档概述

本文档明确FinRag4j项目中Java和Python服务的技术分工边界，确保架构清晰、职责明确。

**文档版本**: v1.0.0

---

## 1. 架构原则

### 1.1 核心原则

**Java是系统唯一业务核心**

- 承担所有业务逻辑
- 管理数据持久化
- 对接前端和外部系统
- 协调各服务组件

**Python是纯工具服务**

- 无数据库连接
- 无业务逻辑
- 仅提供文本处理能力
- 通过HTTP接口调用

### 1.2 分工边界图

```
┌─────────────────────────────────────────────────────────────┐
│                      前端 Vue3                              │
└─────────────────────────────────────────────────────────────┘
                            ↓ HTTP API

│                    Java 业务核心                            │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐ │
│  │  用户权限   │  知识库管理  │  RAG问答    │  Agent业务  │ │
│  └─────────────┴─────────────┴─────────────┴─────────────┘ │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐ │

         ↓ HTTP调用              ↓ HTTP调用          ↓ HTTP调用

└─────────────────┘   └─────────────────┘   └─────────────────┘
```

## 2. Java服务职责

### 2.1 业务逻辑层

| 模块 | 职责 | 说明 |
|------|------|------|
| 用户权限 | 用户认证、角色权限、部门管理 | RBAC完整实现 |
| 知识库管理 | 知识库CRUD、文档绑定、标签分类 | 业务管理 |
| 文档管理 | 文档元数据、版本管理、回收站 | 生命周期管理 |
| RAG问答 | 向量检索、结果重排、回答生成 | 核心业务 |
| Agent业务 | 信贷抽取、合规自查、制度咨询 | 金融垂直业务 |
| 工作流引擎 | 流程编排、任务调度、执行监控 | 流程自动化 |
| 审计日志 | 操作审计、日志存储、报表导出 | 合规审计 |

### 2.2 数据持久化层

| 存储 | 用途 | Java职责 |
|------|------|---------|
| PostgreSQL | 业务数据 | 完全管理 |
| PGVector | 向量数据 | 完全管理 |
| Redis | 缓存/会话 | 完全管理 |
| MinIO | 文件存储 | 元数据管理 |

### 2.3 外部集成层

| 集成 | 方向 | Java职责 |
|------|------|---------|
| 前端 | 提供API | REST接口 |
| Python服务 | 调用方 | HTTP客户端 |
| 大模型服务 | 调用方 | HTTP客户端 |
| 第三方系统 | 提供API | OpenAPI |

## 3. Python服务职责

### 3.1 文档解析

```python
# Python服务仅提供解析能力
POST /parse
{
    "file_path": "/tmp/document.pdf",
    "parse_type": "pdf"
}

# 返回解析结果
{
    "text": "文档文本内容...",
    "metadata": {
        "page_count": 10,
        "word_count": 5000
    }
}
```

### 3.2 OCR识别

```python
# 图片文字识别
POST /ocr
{
    "image_path": "/tmp/image.png"
}

# 返回识别结果
{
    "text": "识别的文字...",
    "confidence": 0.95
}
```

### 3.3 文本分块

```python
# 文本分块处理
POST /chunk
{
    "text": "长文本内容...",
    "chunk_size": 500,
    "overlap": 50
}

# 返回分块结果
{
    "chunks": [
        {"id": 1, "text": "分块1..."},
        {"id": 2, "text": "分块2..."}
    ]
}
```

### 3.4 约束说明

**Python服务禁止**:
- ❌ 连接数据库
- ❌ 实现业务逻辑
- ❌ 存储状态数据
- ❌ 直接对接前端
- ❌ 调用外部服务

**Python服务允许**:
- ✅ 文档解析
- ✅ OCR识别
- ✅ 文本分块
- ✅ 格式转换

## 4. 调用边界示例

### 4.1 文档上传流程

```
用户上传文档
    ↓
Java接收文件
    ↓
Java存储到MinIO
    ↓
Java创建文档元数据
    ↓
Java调用Python解析
    ↓
Python返回解析结果
    ↓
Java调用Python分块
    ↓
Python返回分块结果
    ↓
Java调用大模型向量化
    ↓
Java存储向量到PGVector
    ↓
Java更新文档状态
```

### 4.2 RAG问答流程

```
用户提问
    ↓
Java接收问题
    ↓
Java调用大模型向量化问题
    ↓
Java从PGVector检索相似文档
    ↓
Java重排检索结果
    ↓
Java拼接Prompt
    ↓
Java调用大模型生成回答
    ↓
Java保存对话记录
    ↓
返回回答给用户
```

## 5. 技术选型理由

### 5.1 Java作为业务核心

- **企业级生态**: Spring Boot成熟稳定
- **类型安全**: 编译期检查，减少运行时错误
- **性能优秀**: JVM优化成熟
- **人才充足**: Java开发者众多
- **集成方便**: 丰富的中间件支持

### 5.2 Python作为工具服务

- **AI生态**: NLP工具库丰富
- **快速开发**: 脚本语言灵活
- **资源隔离**: 独立进程，不影响主服务
- **弹性伸缩**: 可独立扩展

## 6. 开发规范

### 6.1 Java开发规范

- 使用Spring Boot 3.2
- 使用MyBatis-Plus操作数据库
- 使用HTTP客户端调用Python服务
- 业务逻辑全部在Service层实现
- Controller仅负责参数校验和响应

### 6.2 Python开发规范

- 使用FastAPI框架
- 不连接任何数据库
- 不实现业务逻辑
- 仅提供文本处理接口
- 接口参数和返回值标准化
